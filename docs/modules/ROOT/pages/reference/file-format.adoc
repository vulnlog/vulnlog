= The Vulnlog file format
:page-aliases: format-overview.adoc

A Vulnlog file is a YAML document that tracks software vulnerabilities reported by one or more SCA scanners, together with the human decisions about them.
Each Vulnlog file is scoped to a single release branch; when the branch reaches end of life, its file becomes a natural historical archive.
From this file, the Vulnlog CLI generates machine-readable outputs such as scanner suppression files and HTML reports.

This page defines the file-level contract; xref:concepts/data-model.adoc[The Vulnlog data model] explains the model and its design principles.

== File name convention

Vulnlog files use the naming convention `<project-name>.vl.yaml`.
The `.vl.` infix identifies Vulnlog files and enables automatic JSON Schema association in IDEs via glob pattern matching.
Simple projects may use `vulnlog.yaml` (see also xref:guides/organize-files-across-branches.adoc[]).

The file location within the repository is not prescribed.
The CLI requires an explicit file path as a positional argument; it does not search for files automatically.

== Schema reference

To enable IDE validation and autocompletion, add a schema reference comment as the first line:

[source,yaml]
----
# $schema: https://vulnlog.dev/schema/vulnlog-v1.json
----

The path can point to a local bundled schema file or to the published JSON schema URL.
The header is optional: it only enables editor support and has no effect on how the CLI reads a file.
A file-writing command keeps the header when the file already has one and leaves it out otherwise; only xref:reference/cli-init.adoc[init] adds it to a newly created file.

== Canonical formatting

Vulnlog files are canonical: every file-writing command (xref:reference/cli-fmt.adoc[fmt], xref:reference/cli-add.adoc[modify add], xref:reference/cli-copy.adoc[modify copy], xref:reference/cli-init.adoc[init]) rewrites the whole file in the same style, derived only from the file's data.
Any valid YAML layout is accepted as input, which makes the commands safe for automation: a file written by hand or by another tool is normalized on the next write.

YAML comments are not part of the format and are not preserved; commands warn when they drop comments.
Record prose in the schema fields designed for it: `comment` and `analysis` on a vulnerability, `description` on a tag, and `note` on a release.
This keeps every remark visible to the outputs generated from the file, such as reports and suppression files.

== File structure

[source,yaml]
----
# $schema: https://vulnlog.dev/schema/vulnlog-v1.json
schemaVersion: "1"

project:
  # Project metadata

tags:
  # Tag definitions (optional)

releases:
  # Release definitions (chronological order, oldest first)

vulnerabilities:
  # Vulnerability entries (newest first)
----

Required top-level sections: `schemaVersion`, `project`, `releases`, `vulnerabilities`.
Optional top-level sections: `tags`.

`vulnerabilities` may be an empty array.

=== Ordering rules

Releases:: Must be in chronological order (oldest first).
This ordering has functional significance -- the CLI uses array position to resolve `--release` range filtering.
A release at position N includes all releases at positions 0 through N.

Vulnerabilities:: Should be in reverse chronological order (newest first).
The most recent findings appear at the top of the file for immediate visibility.
This is a readability convention enforced by the linter, with no functional impact on CLI output.

=== Unique identifiers

All vulnerability identifiers must be unique within a Vulnlog file.
No two vulnerability entries may share the same `id`.
An `id` of one entry must not appear in the `aliases` of another entry, and vice versa.
This ensures unambiguous lookup by any identifier.

== Branching strategy

For projects with a single release line, a single `vulnlog.yaml` file is sufficient.
For projects maintaining multiple release branches in parallel, see the xref:guides/organize-files-across-branches.adoc[Branching Strategy] guide for detailed options and trade-offs.
