== What is Vulnlog?

Vulnlog is a tool for tracking, analyzing, and managing software vulnerabilities reported by SCA (Software Composition Analysis) scanners.

It combines a simple YAML file format with a CLI that generates suppression files, HTML reports, and OpenVEX statements.
The name combines "vulnerability" and "log" -- Vulnlog is a persistent, version-controlled record of your team's vulnerability analysis.

For the full YAML format reference, see the link:vulnlog-spec.adoc[Vulnlog Specification].

=== Features

Vulnerability tracking::
Record and manage vulnerability findings from multiple SCA scanners in a single YAML file.
Each finding captures the scanner report, affected packages, and the team's triage verdict.

Verdict-driven workflow::
Classify vulnerabilities with a clear verdict: `affected`, `not_affected`, or `risk_acceptable`.
Entries without a verdict are tracked as under investigation.
Resolutions (dependency updates, patches) are recorded independently of the verdict.

Flexible tagging::
Tag vulnerability entries with a team-defined taxonomy.
Tags can represent deliverables (`binary`, `container`), usage contexts (`build-dep`, `test-infra`), or any other categorization.

Suppression file generation::
Generate scanner-specific suppression files (`.trivyignore`, `.snyk`, etc.) from the Vulnlog file.
Vulnerabilities with a resolution are excluded from suppression to verify the fix is complete.

HTML report generation::
Generate self-contained HTML reports for communicating vulnerability status to team members, partners, and customers.

OpenVEX statement generation::
Generate OpenVEX documents per release for standards-compliant vulnerability communication.
Verdicts and resolutions map directly to OpenVEX statuses.

Multi-scanner support::
Track findings from multiple SCA scanners in a single file.
Switch scanners without losing analysis history.

Non-scanner sources::
Record vulnerabilities from vendor advisories (e.g., Oracle CPU), penetration tests, and other non-scanner sources alongside scanner findings.

IDE integration::
JSON Schema published to SchemaStore provides validation, autocompletion, and inline documentation.

Git-native::
Vulnlog files are plain YAML designed for version control.
Changes are tracked via Git history.
Each release branch has its own file, which archives naturally at end of life.

Validation and linting::
Validate Vulnlog files against structural rules and cross-reference integrity.
Lint for consistent field ordering and formatting.

CI/CD integration::
The CLI runs on developer machines and in CI pipelines.
Generate suppression files in CI to keep scanner pipelines green for triaged findings.

=== When to Use Vulnlog

Vulnlog is most valuable when your team:

- Uses multiple SCA scanners and needs a unified view of findings.
- Maintains multiple release branches and needs to track vulnerability decisions per branch.
- Needs to communicate vulnerability status via HTML reports or VEX statements.
- Wants to keep vulnerability analysis in the repository, next to the code.
- Wants to prevent stale suppression files that shadow real findings.
- Wants to avoid losing vulnerability analysis when moving to another SCA tool.

Vulnlog may be more than you need if you have a single scanner on a single branch -- in that case, the scanner's native suppression mechanism may suffice.

=== How Vulnlog Compares

==== vs. Scanner-native suppression files

Scanner-native suppression files (`.trivyignore`, `.snyk`) track *what to suppress* but not *why*.
Vulnlog generates these files from a richer decision record, keeping the *why* alongside the *what*.

==== vs. Scanner vendor UIs

Scanner vendor UIs (Snyk dashboard, GitHub Security tab, or Semgrep UI) create vendor lock-in: your triage decisions live in the vendor's platform.
Vulnlog keeps your analysis in a YAML file you own.

==== vs. Ticketing systems, spreadsheets, and Confluence pages

Spreadsheets and wiki pages are disconnected from the code.
Vulnlog lives in the repository and generates suppression files, reports, and VEX documents directly from the data.

=== How Vulnlog Relates to Other Standards

VEX (Vulnerability Exploitability eXchange):: Vulnlog generates OpenVEX documents. Vulnlog is the decision record; VEX is one of its outputs.

SBOM (Software Bill of Materials):: Complementary. SBOMs describe what components are in your software. Vulnlog tracks decisions about vulnerability findings in those components.

SARIF (Static Analysis Results Interchange Format):: A future Vulnlog output for integration with development tools.

== Getting Started

=== Creating a Vulnlog File

[source,bash]
----
vulnlog init --project "My App" --organization "Acme Corp" --author "Security Team" -o my-app.vl.yaml
----

.File Name Convention
[NOTE]
====
For simple projects, `vulnlog.[yaml|yml]` are valid file names.
For multiple Vulnlog files, use `<project-name>.vl.[yaml|yml]`.

=== IDE Setup

Vulnlog's JSON Schema is published to https://www.schemastore.org/[SchemaStore].
IDEs with YAML support can automatically provide validation and autocompletion for `.vl.[yaml|yml]` and `vulnlog.[yaml|yml]` files.

To reference a local schema, add this as the first line of your Vulnlog file:

[source,yaml]
----
# $schema: .vulnlog/vulnlog-schema.json
----

=== CI Pipeline Integration

Example GitHub Actions workflow:

[source,yaml]
----
jobs:
  security-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Vulnlog CLI
        run: # install step

      - name: Generate Trivy suppression file
        run: vulnlog suppress my-app.vl.yaml --reporter trivy --release ${{ github.ref_name }}

      - name: Run Trivy scan
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: fs
          trivyignores: .trivyignore

      - name: Generate vulnerability report
        if: always()
        run: vulnlog report my-app.vl.yaml --release ${{ github.ref_name }} -o vulnlog-report.html

      - name: Upload report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: vulnerability-report
          path: vulnlog-report.html
----

== Common Workflows

=== New finding -- triage later

[source,yaml]
----
  - id: CVE-2026-5678
    releases: [ 2.0.0 ]
    packages: [ "pkg:npm/image-lib@3.1.0" ]
    reports:
      - reporter: trivy
    tags: [ webapp ]
----

=== Adding the same finding to multiple branch files

[source,bash]
----
vulnlog add branch-8.1.vl.yaml branch-8.2.vl.yaml branch-8.3.vl.yaml \
  --vuln-id CVE-2026-5678 --reporter trivy
----

=== Finding does not affect us

[source,yaml]
----
  - id: CVE-2026-5678
    releases: [ 2.0.0 ]
    description: Memory corruption in image-lib
    packages: [ "pkg:npm/image-lib@3.1.0" ]
    reports:
      - reporter: trivy
        at: 2026-03-01
    tags: [ webapp ]
    analysis: >
      The vulnerable code path requires SVG input processing.
      Our application only accepts PNG and JPEG formats.
    analyzed_at: 2026-03-02
    verdict: not_affected
    justification: vulnerable_code_not_in_execute_path
----

=== Not affected, but dependency updated anyway

[source,yaml]
----
  - id: CVE-2026-5678
    releases: [ 2.0.0 ]
    description: Memory corruption in image-lib
    packages: [ "pkg:npm/image-lib@3.1.0" ]
    reports:
      - reporter: trivy
        at: 2026-03-01
    tags: [ webapp ]
    analysis: >
      The vulnerable code path requires SVG input processing.
      Our application only accepts PNG and JPEG formats.
    verdict: not_affected
    justification: vulnerable_code_not_in_execute_path
    resolution:
      in: 2.0.1
      note: "Updated image-lib from 3.1.0 to 3.2.0 during routine upgrade"
----

=== Affected and fixed in the next release

[source,yaml]
----
  - id: CVE-2026-5678
    releases: [ 2.0.0 ]
    description: Memory corruption in image-lib
    packages: [ "pkg:npm/image-lib@3.1.0" ]
    reports:
      - reporter: trivy
        at: 2026-03-01
    tags: [ webapp ]
    analysis: >
      Confirmed exploitable via crafted PNG uploads.
    analyzed_at: 2026-03-02
    severity: high
    verdict: affected
    resolution:
      in: 2.0.1
      at: 2026-03-03
      ref: "https://jira.example.com/browse/SEC-99"
      note: "Updated image-lib from 3.1.0 to 3.2.0"
----

=== Risk accepted with permanent suppression

[source,yaml]
----
  - id: CVE-2026-5678
    releases: [ 2.0.0 ]
    description: Memory corruption in image-lib
    packages: [ "pkg:npm/image-lib@3.1.0" ]
    reports:
      - reporter: trivy
        suppress: {}
    tags: [ webapp ]
    analysis: >
      Vulnerability confirmed but only exploitable with
      specially crafted input that our validation layer rejects.
    severity: medium
    verdict: risk_acceptable
    comment: >
      Risk accepted by project lead. Revisit when upstream
      releases a stable fix.
----

=== Internal dependency, no customer impact

[source,yaml]
----
  - id: CVE-2026-9999
    releases: [ 2.0.0 ]
    description: Command injection in build-tool
    packages: [ "pkg:npm/build-tool@5.0.0" ]
    reports:
      - reporter: trivy
        at: 2026-03-01
    tags: [ build-dep ]
    analysis: >
      Affects the build pipeline only. The vulnerable command
      injection requires untrusted input which the CI environment
      does not expose.
    verdict: not_affected
    justification: vulnerable_code_cannot_be_controlled_by_adversary
----

== Managing Multiple Release Branches

Maintain one Vulnlog file per release branch, all on the main branch:

[source]
----
my-app-8.1.vl.yaml    # Release branch 8.1.x
my-app-8.2.vl.yaml    # Release branch 8.2.x
my-app-8.3.vl.yaml    # Release branch 8.3.x
----

Use `vulnlog add` to create stub entries across multiple files at once.

== CLI Reference

=== Usage

[source]
----
vulnlog <command> <file> [flags]
----

Every command except `init` requires a Vulnlog file as a positional argument.

=== Filtering

Common filters for commands: `stats`, `report`, `suppress` and `vex`.

[cols="2,3"]
|===
| Flag | Description

| `--release <id>`
| Include vulnerabilities affecting all releases up to and including this release.

| `--tags <id>`
| Filter by tag (repeatable).
|===

=== Commands

==== `vulnlog init`

[source,bash]
----
vulnlog init -o my-app.vl.yaml --project "My App" --organization "Acme Corp" --author "Security Team"
----

==== `vulnlog validate`

[source,bash]
----
vulnlog validate my-app.vl.yaml
vulnlog validate my-app.vl.yaml --strict
----

==== `vulnlog lint`

[source,bash]
----
vulnlog lint my-app.vl.yaml          # Check only
vulnlog lint my-app.vl.yaml --fix    # Rewrite with corrected ordering
----

==== `vulnlog stats`

[source,bash]
----
vulnlog stats my-app.vl.yaml
vulnlog stats my-app.vl.yaml --release 8.1.1
----

==== `vulnlog report`

[source,bash]
----
vulnlog report my-app.vl.yaml
vulnlog report my-app.vl.yaml --release 8.1.1 -o report.html
----

==== `vulnlog suppress`

[source,bash]
----
vulnlog suppress my-app.vl.yaml --reporter trivy --release 8.1.1
vulnlog suppress my-app.vl.yaml
vulnlog suppress my-app.vl.yaml --reporter trivy -o custom/.trivyignore
----

==== `vulnlog vex`

[source,bash]
----
vulnlog vex my-app.vl.yaml --release 8.1.1
vulnlog vex my-app.vl.yaml --release 8.1.1 --tags binary -o vex-binary.json
----

==== `vulnlog add`

[source,bash]
----
vulnlog add branch-8.1.vl.yaml branch-8.2.vl.yaml branch-8.3.vl.yaml \
  --vuln-id CVE-2026-001 --reporter trivy
----

=== Exit Codes

[cols="1,1,3"]
|===
| Code | Name | Description

| `0` | Success | Command completed successfully.
| `1` | General error | Unexpected error.
| `2` | Validation error | The Vulnlog file contains validation errors.
| `3` | File not found | The specified file does not exist.
| `4` | Lint violation | The file has lint violations (without `--fix`).
| `5` | Missing required flag | A required flag was not provided.
| `6` | Invalid flag value | A flag references an unknown entity or unsupported type.
|===
