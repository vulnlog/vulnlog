== Overview

Vulnlog is a YAML-based format for tracking, analyzing, and managing software vulnerabilities reported by one or multiple Software Composition Analysis (SCA) scanners.
Each Vulnlog file is scoped to a single release branch.
The lifecycle of the file follows the lifecycle of the release branch -- when a release branch reaches end of life (EoL), its Vulnlog file becomes a natural historical archive.

A Vulnlog file serves as a human-authored decision ledger.
From this file, the Vulnlog CLI generates machine-readable outputs: SCA scanner suppression files, HTML reports, and OpenVEX statements.

=== Scope

Vulnlog is focused on security vulnerabilities reported by SCA scanners.
License compliance findings, configuration issues, and other informational findings are out of scope for this version of the specification.

=== Design Principles

Compact and readable:: The format prioritizes ease of writing and reading over exhaustive detail.
The common case -- reporting a finding and recording a verdict -- should require minimal boilerplate.
IDE-supported:: A JSON Schema published to SchemaStore provides validation, autocompletion, and inline documentation in editors.
Git-friendly:: The file is designed for version control.
Changes are tracked via Git history rather than in-file audit trails.
Data-oriented:: Entries are plain data records with no behavior.
The CLI applies transformations and business logic.

=== File Name Convention

Vulnlog files use the naming convention `<project-name>.vl.[yaml|yml]`.
The `.vl.` infix identifies Vulnlog files and enables automatic JSON Schema association in IDEs via glob pattern matching.
Simple projects may use `vulnlog.[yaml|yml]`.

The file location within the repository is not prescribed.
The CLI requires an explicit file path as a positional argument; it does not search for files automatically.

=== Schema Reference

To enable IDE validation and autocompletion, add a schema reference comment as the first line of the Vulnlog file:

[source,yaml]
----
# $schema: .vulnlog/vulnlog-schema.json
----

The path can point to a local bundled schema file or to the published SchemaStore URL.

=== Multi-Branch Workflow

The recommended approach for teams managing multiple release branches is to maintain multiple Vulnlog files on the main branch, one per release branch:

[source]
----
demo-project-8.1.vl.yaml    # Release branch 8.1.x
demo-project-8.2.vl.yaml    # Release branch 8.2.x
demo-project-8.3.vl.yaml    # Release branch 8.3.x
----

Each file is self-contained.
The CLI operates on one file at a time.
This approach avoids cherry-picking vulnerability analysis to older Git branches.

It remains possible to maintain a Vulnlog file directly on each release branch.
Both approaches can coexist.

See <<vulnlog-add>> for CLI support to add entries across multiple files, and <<Cross-File References>> for the planned `ref` mechanism to share analysis between files.

== File Structure

[source,yaml]
----
# $schema: .vulnlog/vulnlog-schema.json
schemaVersion: "1"

project:
  # Project metadata

tags:
  # Tag definitions (optional)

releases:
  # Release definitions (chronological order, oldest first)

vulnerabilities:
  # Vulnerability entries (newest first)
----

Required top-level sections: `schemaVersion`, `project`, `releases`, `vulnerabilities`.
Optional top-level sections: `tags`.

`vulnerabilities` may be an empty array.

=== Ordering Rules

Releases:: Must be in chronological order (oldest first). This ordering has functional significance -- the CLI uses array position to resolve `--release` range filtering. A release at position N includes all releases at positions 0 through N.

Vulnerabilities:: Should be in reverse chronological order (newest first). The most recent findings appear at the top of the file for immediate visibility. This is a readability convention enforced by the linter, with no functional impact on CLI output.

=== Unique Identifiers

All vulnerability identifiers must be unique within a Vulnlog file.
No two vulnerability entries may share the same `id`.
An `id` of one entry must not appear in the `aliases` of another entry, and vice versa.
This ensures unambiguous lookup by any identifier.

Project-level metadata.
Used in report generation and as author information in VEX documents.

[cols="1,1,1,3"]
|===
| Field | Type | Required | Description

| `organization`
| String
| Yes

| `name`
| String
| Yes

| `author`
| String
| Yes
| Name of the responsible security team or author. Used as the author in VEX documents.

| `contact`
| String (email)
| No
| Contact email for the security team. Used as the author contact in VEX documents. This can be the contact information of your https://securitytxt.org/[security.txt].
|===

=== `tags`

Optional.
Defines labels for categorizing and scoping vulnerability entries.
Tags provide a flexible, team-defined taxonomy that can represent customer-facing deliverables (`binary`, `container`, `sdk`), internal usage contexts (`build-dep`, `test-infra`), or any other categorization.

[cols="1,1,1,3"]
|===
| Field | Type | Required | Description

| `id`
| String
| Yes
| Unique identifier for the tag. Referenced by vulnerability entries and release purl entries.

| `description`
| String
| No
| Human-readable description of what this tag represents.
|===

=== `releases`

Defines the releases tracked in this Vulnlog file.
Releases must be listed in chronological order (oldest first).
This ordering is used by the CLI to resolve `--release` filtering.

[cols="1,1,1,3"]
|===
| Field | Type | Required | Description

| `id`
| String
| Yes
| Unique release identifier (e.g., `8.1.1`, `2.0.0-SNAPSHOT`).

| `published_at`
| String (date, `YYYY-MM-DD`)
| No
| Publication date of the release. Absence indicates the release is not yet published.

| `purls`
| Array of <<purl-entry>>
| No
| Versioned Package URLs identifying the release artifacts. Used as product identifiers in VEX documents.
|===

[[purl-entry]]
==== PURL Entry

A release artifact identified by its Package URL, optionally tagged for scoping.

[cols="1,1,1,3"]
|===
| Field | Type | Required | Description

| `purl`
| String (Package URL)
| Yes
| The versioned Package URL for this artifact in this release.

| `tags`
| Array of String
| No
| Tags associated with this purl. Used to match vulnerability tags for scoped VEX generation. Must reference tags defined in the `tags` section.
|===

When a vulnerability entry has tags that match a purl's tags, only the matching purls are included as products in the VEX statement.
Purls without tags are included in VEX statements regardless of vulnerability tags.

=== `vulnerabilities`

Each entry represents a single vulnerability finding and the human verdict about it.

NOTE: The verdict should reflect the most severe impact across all listed packages.
If a package appears in multiple contexts, the analysis should focus on the highest-risk context.

[cols="1,1,1,3"]
|===
| Field | Type | Required | Description

| `id`
| String
| Yes
| Primary identifier. Preferably a CVE ID. May be a scanner-specific ID if no CVE exists. Must be unique across all entries and aliases.

| `name`
| String
| No
| Common name if one exists (e.g., "Log4Shell", "Heartbleed"). Displayed prominently in reports.

| `aliases`
| Array of String
| No
| Alternative identifiers (e.g., GHSA IDs, scanner-specific IDs). Must not overlap with any `id` or `aliases` in other entries.

| `releases`
| Array of String
| Yes
| Release identifiers affected. Must reference IDs from the `releases` section.

| `description`
| String
| No
| Brief human-readable description. Optional to allow quick stub entries during triage.

| `packages`
| Array of String (Package URL)
| Yes
| Package URLs of affected dependencies.

| `reports`
| Array of <<report-entry>>
| Yes
| Scanner reports that identified this vulnerability.

| `tags`
| Array of String
| No
| Tags for categorizing and scoping. Must reference tags defined in the `tags` section. Used for CLI filtering and VEX product scoping.

| `analysis`
| String
| No
| Free-text analysis and rationale for the triage decision. Maps to `impact_statement` in VEX for `not_affected` and `action_statement` for `risk_acceptable`.

| `analyzed_at`
| String (date, `YYYY-MM-DD`)
| No
| Date the analysis was performed. Must not be before the earliest report date.

| `comment`
| String
| No
| Additional information not related to the analysis itself. May include evidence references, methodology notes, decision-making context, or reminders for future review. Not used in any output generation.

| `severity`
| String (enum)
| Conditional
| Severity assessment. Required when `verdict` is `affected` or `risk_acceptable`. See <<severity-values>>.

| `verdict`
| String (enum)
| No
| The human triage decision. See <<verdict-values>>. Absence indicates under investigation.

| `justification`
| String (enum)
| Conditional
| Required when `verdict` is `not_affected`. See <<justification-values>>.

| `resolution`
| <<resolution-entry>>
| No
| Records that the underlying dependency was updated or the vulnerability was otherwise addressed. Independent of the verdict. See <<resolution-entry>>.
|===

Records that a dependency update or other corrective action was taken, independent of the verdict.
A vulnerability may be `not_affected` (the verdict) yet still have a `resolution` (the dependency was updated defensively).
A vulnerability may be `affected` with a `resolution` indicating it was fixed in a subsequent release.

[cols="1,1,1,3"]
|===
| Field | Type | Required | Description

| `in`
| String
| Yes
| Release in which the resolution was applied. Must reference a release ID from the `releases` section.

| `at`
| String (date, `YYYY-MM-DD`)
| No
| Date the resolution was applied.

| `ref`
| String (URI)
| No
| Reference to the issue or ticket tracking the resolution.

| `note`
| String
| No
| Brief description of how the vulnerability was resolved.
|===

==== Report Entry

[cols="1,1,1,3"]
|===
| Field | Type | Required | Description

| `reporter`
| String (enum)
| Yes
| The scanner or source type. See <<reporter-type-values>>.

| `at`
| String (date, `YYYY-MM-DD`)
| No
| Date the vulnerability was reported. When omitted, the CLI uses the current date during output generation.

| `source`
| String
| No
| Description of the report source. Optional for scanner reporters. Required by the CLI for reporter type `other`.

| `vuln_id`
| Array of String
| No
| Scanner-specific vulnerability identifiers. Used for scanner-specific suppression files.

| `suppress`
| <<suppress-entry>>
| No
| Scanner-specific suppression configuration. When present without `expires_at`, the suppression is permanent.

[[suppress-entry]]
==== Suppress Entry

[cols="1,1,1,3"]
|===
| Field | Type | Required | Description

| `expires_at`
| String (date, `YYYY-MM-DD`)
| No
| Expiration date. After this date, the entry is excluded from the generated suppression file. When omitted, the suppression is permanent.
|===

== Enumerations

[[verdict-values]]
=== Verdict Values

The term "verdict" is used deliberately to distinguish from automated scanner statuses -- it reflects a deliberate human judgment about the impact of a vulnerability.

[cols="1,3,2"]
|===
| Value | Description | OpenVEX Mapping

| `affected`
| The vulnerability affects this release. Requires `severity`. When a `resolution` is present, maps to VEX `fixed`.
| `affected` (without resolution) or `fixed` (with resolution)

| `not_affected`
| The vulnerability does not affect this release. Requires `justification`.
| `not_affected`

| `risk_acceptable`
| The vulnerability affects this release but the risk has been assessed and accepted. Requires `severity`.
| `affected` (with `action_statement` from `analysis`)

| _(absent)_
| Not yet triaged.
| `under_investigation`
|===

NOTE: "Fixed" is not a verdict. Fixing a vulnerability is a resolution action, not a judgment about impact. Use `verdict: affected` with a `resolution` block to record that a vulnerability was real and subsequently fixed. Use `verdict: not_affected` with an optional `resolution` to record that a vulnerability was not relevant but the dependency was updated anyway.

[[severity-values]]
=== Severity Values

Required when `verdict` is `affected` or `risk_acceptable`.

[cols="1,3"]
|===
| Value | Description

| `critical`
| Immediate action required. Exploitable with severe impact.

| `high`
| Action required. Significant impact if exploited.

| `medium`
| Action recommended. Moderate impact.

| `low`
| Low priority. Minimal impact or difficult to exploit.
|===

[[justification-values]]
=== Justification Values (OpenVEX)

[cols="1,3"]
|===
| Value | Description

| `component_not_present`
| The vulnerable component is not included in the deliverable.

| `vulnerable_code_not_present`
| The vulnerable code is present but the specific vulnerable function or code path is not included.

| `vulnerable_code_not_in_execute_path`
| The vulnerable code is present but cannot be reached during execution.

| `vulnerable_code_cannot_be_controlled_by_adversary`
| The vulnerable code is reachable but cannot be triggered by an attacker.

| `inline_mitigations_already_exist`
| The vulnerability is mitigated by existing controls (e.g., WAF, input validation).
|===

== Defaults and Implicit Behavior

Omitted `tags` section:: The file has no tag definitions. VEX generation includes all purls as products.
Omitted `verdict`:: Treated as `under_investigation`. The CLI warns about entries without a verdict.
Omitted `reports[].at`:: The CLI uses the current date and time during output generation.
Omitted `suppress.expires_at`:: The suppression is permanent (never expires).
Omitted `analyzed_at`:: No analysis date is recorded.
Omitted `vulnerabilities[].tags`:: The vulnerability is not scoped. Included in all unfiltered outputs.
Omitted `releases[].purls[].tags`:: The purl is included in VEX statements regardless of vulnerability tags.

== Suppression Behavior

=== Suppression Rules

Entries where `verdict` is `not_affected` are always included in suppression files.

Entries where `verdict` is `affected` with a `resolution` are *not* included in suppression files.
The vulnerable dependency version should no longer be present after the resolution.
If the scanner still reports the finding, the resolution is incomplete.

For all other cases (`affected` without resolution, `risk_acceptable`, absent), suppression requires an explicit `suppress` block on the report entry.
If the `suppress` block includes `expires_at`, the suppression is temporary.
If `expires_at` is omitted, the suppression is permanent.

// tag::dev-notes[]
////
IMPLEMENTATION NOTE:

Suppression generation algorithm:

1. Determine target reporter types:
   - If `--reporter <type>`, use that type.
   - If omitted, generate for all reporter types found in vulnerability reports.
2. Determine target releases:
   - If `--release <id>`, include all releases up to and including that release.
   - If omitted, include all releases.
3. Filter vulnerabilities to those affecting target releases.
4. If `--tags` specified, further filter to vulnerabilities with matching tags.
5. For each target reporter type:
   a. Collect vulnerability entries with a report for that type.
   b. Evaluate suppression eligibility:
      - `verdict` is `not_affected` -> INCLUDE.
      - `verdict` is `affected` AND `resolution` present -> EXCLUDE.
      - Otherwise: INCLUDE only if `suppress` block exists AND
        (`expires_at` is absent OR `expires_at` >= today).
   c. Determine suppression identifier:
      - `reports[].vuln_id` if present -> use those IDs.
      - Otherwise -> `vulnerabilities[].id`.
      - `aliases` are NOT used.
   d. Determine output path: `--output` flag or default path for the reporter type.
   e. Apply scanner-specific formatting.
   f. For formats supporting comments, include a back-reference to the Vulnlog file and entry.
////
// end::dev-notes[]

=== Suppression Identifier Resolution

. `vuln_id` values from the report entry for the target reporter type, if present.
. The vulnerability's `id` field.

`aliases` are not used for suppression.

== VEX Generation

OpenVEX documents are generated via the CLI.
Vulnlog tags + release purls map to OpenVEX products.
Vulnlog packages map to OpenVEX subcomponents.

=== Field Mapping

.Minimal required OpenVEX fields mapped to Vulnlog fields.
[cols="2,2"]
|===
| OpenVEX Field | Vulnlog Source

| `@context`
| Static value: `https://openvex.dev/ns/v0.2.0`

| `@id`
| Generated IRI: `https://vulnlog.dev/vex/<UUID>`

| `author`
| `project.author`. When `project.contact` is present: `"<author> (<contact>)"`.

| Document generation timestamp.

| `version`
| `1` for new documents. Previous version + 1 when updating an existing document (see <<VEX Document Versioning>>).

| `statements[].vulnerability.name`

| `statements[].status`
| Derived from `verdict` and `resolution` (see <<VEX Status Mapping>>)

| `statements[].justification`
| `vulnerabilities[].justification`

| `statements[].impact_statement`
| `vulnerabilities[].analysis` (when verdict is `not_affected`)
|===

=== Product Scoping via Tags

When a vulnerability has tags, the VEX generator matches those tags against purl entries on the target release.
Only purls with matching tags are included as products.
Purls without tags are always included.
Vulnerabilities without tags are associated with all purls.

[[vex-document-versioning]]
=== VEX Document Versioning

The OpenVEX `version` field is an incrementing integer.
Since the CLI is stateless, the version is determined by reading the existing output file:

. The CLI reads the file at the output path (specified by `-o` or the default `vex.json`).
. If the file exists and is a valid OpenVEX document, the CLI extracts the current `version`.
. The new document is generated with `version` = previous version + 1.
. If no existing file is found, the document starts at version `1`.
. The CLI writes the new document only if the statements have changed compared to the previous document. If unchanged, the file is not written and the CLI reports "No changes detected."

== HTML Report Generation

The primary output is a self-contained HTML file with a summary table.
Each vulnerability entry is a single row.

`name` (if present) is displayed prominently.
`aliases` shown as secondary identifiers.
Tags are displayed for filtering context.
`comment` is not included in HTML reports (it is internal-only metadata).

=== Detail Pages (Future)

A future version will generate per-vulnerability detail pages alongside the summary:

[source]
----
vulnlog-report/
  index.html                    # Summary table with links
  CVE-2021-44228.html           # Detail page
  ...
----

== Validation Rules

. `releases` in vulnerability entries must reference IDs defined in `releases`.
. `tags` in vulnerability entries must reference IDs defined in `tags` (when `tags` section exists).
. `tags` in release purl entries must reference IDs defined in `tags`.
. `resolution.in` must reference a release ID defined in `releases`.
. `reporter` must be a valid reporter type enum value.
. `justification` is required when `verdict` is `not_affected` and forbidden otherwise.
. `severity` is required when `verdict` is `affected` or `risk_acceptable`.
. `source` is required on report entries when `reporter` is `other`.
. All vulnerability `id` values must be unique.
. No `id` may appear in any other entry's `aliases`, and no two entries may share an alias.
. `suppress.expires_at`, when present, must be a valid future or present date at CLI execution time.
. `analyzed_at`, when present, must not be before the earliest `at` date across reports for that vulnerability.
. Entries in `releases` must be in chronological order (oldest first).
. All entries in `packages` must be valid Package URLs.

IMPLEMENTATION NOTE:

Validation enforcement:

Rules 1-4: CLI runtime (cross-reference checks).
Rules 5-7: JSON Schema (enum validation, conditional `if`/`then`/`else`).
Rules 8-14: CLI runtime.

Two-phase validation:
  Phase 1: JSON Schema (structural, fast, IDE-compatible).
  Phase 2: CLI (cross-references, dates, uniqueness, ordering).

Report all errors at once. Group by vulnerability entry.

  Collect all `id` values and all `aliases` values into a single set.
  If any value appears more than once, report a validation error
  identifying both entries that share the identifier.

== CLI Design

=== Usage

[source]
----
vulnlog <command> <file> [flags]
----

Every command except `init` requires a Vulnlog file as a positional argument.
No automatic file detection.

=== Global Flags

[cols="2,3"]
|===
| Flag | Description

| `-q, --quiet`
| Suppress non-essential output.

| `-v, --verbose`
| Verbose output.

| `--no-color`
| Disable colored output.

| `--format <f>`
| Output format for structured output: `yaml` (default). Future: `json`, `text`.

| `--version`
| Print version and exit.

| `--help`
| Print help.
|===

=== Filtering Flags

Available on `stats`, `report`, `suppress`, and `vex` commands.

[cols="2,3"]
|===
| Flag | Description

| `--release <id>`
| Include vulnerabilities affecting all releases up to and including this release.

| `--tags <id>`
| Filter by tag (repeatable).
|===

vulnlog validate <file> [--strict]
----

`--strict` treats warnings as errors (e.g., missing verdict, missing description).

[[vulnlog-lint]]
==== `vulnlog lint`

[source]
----
vulnlog lint <file> [--fix]
----

Enforces:

Top-level section order:: `schemaVersion`, `project`, `tags`, `releases`, `vulnerabilities`.

Vulnerability field order:: `id`, `name`, `aliases`, `releases`, `description`, `packages`, `reports`, `tags`, `analysis`, `analyzed_at`, `comment`, `severity`, `verdict`, `justification`, `resolution`.

Release chronological order:: Oldest first.

Vulnerability reverse chronological order:: Newest first (based on the earliest `reports[].at` date, entries without dates sorted last).

[[vulnlog-add]]
==== `vulnlog add`

Adds a vulnerability stub entry to one or more Vulnlog files.

[source]
----
vulnlog add <file...> --vuln-id <id> --reporter <type> [flags]
----

[cols="2,3"]
|===
| Flag | Description

| `--vuln-id <id>`
| Vulnerability ID to add (required).

| `--reporter <type>`
| Reporter type for the report entry (required).

| `--release <id>`
| Release to associate. When omitted, adds to all releases in each file.
|===

The command inserts a minimal stub entry at the top of the vulnerabilities array in each file.
If an entry with the same `id` already exists in a file, the command skips that file and reports a warning.

Example:

[source,bash]
----
vulnlog add branch-8.1.vl.yaml branch-8.2.vl.yaml branch-8.3.vl.yaml \
  --vuln-id CVE-2026-001 --reporter trivy
----

== Future Directions

=== Cross-File References

A future version will introduce a `ref` field on vulnerability entries to reference an entry in another Vulnlog file.
This addresses the duplication problem in multi-branch setups.

// tag::dev-notes[]
////
DESIGN: Cross-file references via `ref`

A vulnerability entry can use `ref` instead of `id` to reference an entry from another file:

  # branch-8.1.vl.yaml
  vulnerabilities:
    - ref: "branch-8.3.vl.yaml#CVE-2026-001"
      releases: [ 8.1.5 ]
      reports:
        - reporter: trivy
      verdict: not_affected
      justification: vulnerable_code_not_in_execute_path

Rules:
1. `ref` references `id` entries only, never other `ref` entries.
2. References should point from shorter-lived branches to longer-lived branches.
3. The CLI validates references at `validate` time.
4. Required fields in a `ref` entry: `ref`, `releases`, `reports`.
5. `ref` and `id` are mutually exclusive on the same entry.
6. Merge semantics: field-level override. Defined fields override, omitted fields are inherited.
////
// end::dev-notes[]
