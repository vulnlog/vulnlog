1. Overview

Vulnlog is a YAML-based format for tracking, analyzing, and managing software vulnerabilities reported by one or multiple Software Composition Analysis (SCA) scanners. Each Vulnlog file is scoped to a single release branch. The lifecycle of the file follows the lifecycle of the release branch — when a release branch reaches end of life (EoL), its Vulnlog file becomes a natural historical archive.

A Vulnlog file serves as a human-authored decision ledger. From this file, the Vulnlog CLI generates machine-readable outputs: SCA scanner suppression files, HTML reports, and OpenVEX statements.

1.1. Scope

Vulnlog is focused on security vulnerabilities reported by SCA scanners. License compliance findings, configuration issues, and other informational findings are out of scope for this version of the specification.

1.2. Design Principles

Compact and readable

The format prioritizes ease of writing and reading over exhaustive detail. The common case — reporting a finding and recording a verdict — should require minimal boilerplate.

IDE-supported

A JSON Schema published to SchemaStore provides validation, autocompletion, and inline documentation in editors.

Git-friendly

The file is designed for version control. Changes are tracked via Git history rather than in-file audit trails.

Data-oriented

Entries are plain data records with no behavior. The CLI applies transformations and business logic.

1.3. File Convention

Vulnlog files use the naming convention <project-name>.vl.yaml (or .vl.yml). The .vl. infix identifies Vulnlog files and enables automatic JSON Schema association in IDEs via glob pattern matching. Simple projects may use vulnlog.yaml (or vulnlog.yml).

The file location within the repository is not prescribed. The CLI requires an explicit file path as a positional argument; it does not search for files automatically.

1.4. Multi-Branch Workflow

The recommended approach for teams managing multiple release branches is to maintain multiple Vulnlog files on the main branch, one per release branch:

demo-project-8.1.vl.yaml    # Release branch 8.1.x
demo-project-8.2.vl.yaml    # Release branch 8.2.x
demo-project-8.3.vl.yaml    # Release branch 8.3.x

Each file is self-contained. The CLI operates on one file at a time. This approach avoids cherry-picking vulnerability analysis to older Git branches.

It remains possible to maintain a Vulnlog file directly on each release branch. Both approaches can coexist.

2. File Structure

version: "1"

project:
  # Project metadata

tags:
  # Tag definitions (optional)

releases:
  # Release definitions (chronological order)

vulnerabilities:
  # Vulnerability entries

Required top-level sections: version, project, releases, vulnerabilities. Optional top-level sections: tags.

vulnerabilities may be an empty array.

3. Sections

3.1. version

Field Type Required Description

version

String

Yes

Schema version of the Vulnlog file. For this specification it is "1".

3.2. project

Project-level metadata. Used in report generation and as author information in VEX documents.

Field Type Required Description

organization

String

Yes

Name of the organization or vendor.

name

String

Yes

Name of the software project.

security_team

String

No

Name of the responsible security team. Used as the author in VEX documents.

security_team_email

String (email)

No

Contact email for the security team. Used as the author contact in VEX documents.

3.3. tags

Optional. Defines labels for categorizing and scoping vulnerability entries. Tags provide a flexible, team-defined taxonomy that can represent customer-facing deliverables (binary, container, sdk), internal usage contexts (build-dep, test-infra), compliance scopes (pci, hipaa, sox), or any other categorization.

Field Type Required Description

id

String

Yes

Unique identifier for the tag. Referenced by vulnerability entries and release purl entries.

description

String

No

Human-readable description of what this tag represents.

3.4. releases

Defines the releases tracked in this Vulnlog file. Releases must be listed in chronological order (oldest first). This ordering is used by the CLI to resolve --release filtering.

Field Type Required Description

id

String

Yes

Unique release identifier (e.g., 8.1.1, 2.0.0-SNAPSHOT).

published_at

String (date, YYYY-MM-DD)

No

Publication date of the release. Absence indicates the release is not yet published.

purls

Array of PURL Entry

No

Versioned Package URLs identifying the release artifacts. Used as product identifiers in VEX documents.

3.4.1. PURL Entry

A release artifact identified by its Package URL, optionally tagged for scoping.

Field Type Required Description

purl

String (Package URL)

Yes

The versioned Package URL for this artifact in this release.

tags

Array of String

No

Tags associated with this purl. Used to match vulnerability tags for scoped VEX generation. Must reference tags defined in the tags section.

When a vulnerability entry has tags that match a purl’s tags, only the matching purls are included as products in the VEX statement. Purls without tags are included in VEX statements regardless of vulnerability tags.

3.5. vulnerabilities

Each entry represents a single vulnerability finding and the human verdict about it. The verdict reflects the most severe impact across all listed packages.

Field Type Required Description

id

String

Yes

Primary identifier. Preferably a CVE ID. May be a scanner-specific ID if no CVE exists.

name

String

No

Common name if one exists (e.g., "Log4Shell", "Heartbleed"). Displayed prominently in reports.

aliases

Array of String

No

Alternative identifiers (e.g., GHSA IDs, scanner-specific IDs). Used for cross-referencing in reports.

releases

Array of String

Yes

Release identifiers affected. Must reference IDs from the releases section.

description

String

No

Brief human-readable description. Optional to allow quick stub entries during triage.

packages

Array of String (Package URL)

Yes

Package URLs of affected dependencies.

reports

Array of Report Entry

Yes

Scanner reports that identified this vulnerability.

tags

Array of String

No

Tags for categorizing and scoping. Must reference tags defined in the tags section. Used for CLI filtering and VEX product scoping.

analysis

String

No

Free-text analysis and rationale for the triage decision. Maps to impact_statement in VEX for not_affected and action_statement for risk_acceptable.

analyzed_at

String (date, YYYY-MM-DD)

No

Date the analysis was performed. Must not be before the earliest report date.

comment

String

No

Additional information not related to the analysis itself. May include evidence references, methodology notes, decision-making context, or reminders for future review. Not used in any output generation.

severity

String (enum)

Conditional

Severity assessment. Required when verdict is affected or risk_acceptable. See Severity Values.

verdict

String (enum)

No

The human triage decision. See Verdict Values. Absence indicates under investigation.

justification

String (enum)

Conditional

Required when verdict is not_affected. See Justification Values (OpenVEX).

fixed_in

String

Conditional

Required when verdict is fixed. References the release with the fix.

fixed_at

String (date, YYYY-MM-DD)

No

Date the fix was applied. Only meaningful when verdict is fixed.

fix_ref

String (URI)

No

Reference to the issue or ticket tracking the fix.

fix_note

String

No

Brief description of how the vulnerability was fixed.

3.5.1. Report Entry

Field Type Required Description

reporter

String (enum)

Yes

The scanner or source type. See Reporter Type Values.

at

String (date, YYYY-MM-DD)

Yes

Date the vulnerability was reported.

source

String

No

Description of the report source. Optional for scanner reporters. Required by the CLI for reporter type other.

vuln_id

Array of String

No

Scanner-specific vulnerability identifiers. Used for scanner-specific suppression files.

suppress

Suppress Entry

No

Scanner-specific suppression configuration.

3.5.2. Suppress Entry

Field Type Required Description

expires_at

String (date, YYYY-MM-DD)

Yes

Expiration date. After this date, the entry is excluded from the generated suppression file.

4. Enumerations

The term "verdict" is used deliberately to distinguish from automated scanner statuses — it reflects a deliberate human judgment.

4.2. Severity Values

Value Description

Immediate action required. Exploitable with severe impact.

Action required. Significant impact if exploited.

Action recommended. Moderate impact.

5. Defaults and Implicit Behavior

Omitted tags section

The file has no tag definitions. Vulnerability entries cannot use tags. VEX generation includes all purls as products.

Omitted verdict

Treated as under_investigation. The CLI warns about entries without a verdict.

Omitted releases[].published_at

The release is considered unpublished.

Omitted analyzed_at

No analysis date is recorded.

Omitted vulnerabilities[].tags

The vulnerability is not scoped. Included in all unfiltered outputs.

Omitted releases[].purls[].tags

The purl is included in VEX statements regardless of vulnerability tags.

6. Suppression Behavior

6.1. Suppression Rules

Entries where verdict is fixed are never included in suppression files.

Entries where verdict is not_affected are always included.

For all other verdicts (affected, risk_acceptable, absent), suppression requires an explicit, non-expired suppress block on the report entry.

6.2. Suppression Identifier Resolution

  1. vuln_id values from the report entry for the target reporter type, if present.

  2. The vulnerability’s id field.

aliases are not used for suppression.

7. VEX Generation

OpenVEX documents are generated via the CLI. Vulnlog tags + release purls map to OpenVEX products. Vulnlog packages map to OpenVEX subcomponents.

7.1. Field Mapping

OpenVEX Field Vulnlog Source

author

project.security_team and project.security_team_email

statement.vulnerability

vulnerabilities[].id

statement.products

releases[].purls[].purl for the target release, filtered by matching tags

statement.products[].subcomponents

vulnerabilities[].packages[]

statement.status

vulnerabilities[].verdict (mapped per Verdict Values)

statement.justification

vulnerabilities[].justification

statement.impact_statement

vulnerabilities[].analysis (when verdict is not_affected)

statement.action_statement

vulnerabilities[].analysis (when verdict is risk_acceptable)

7.2. Product Scoping via Tags

When a vulnerability has tags, the VEX generator matches those tags against purl entries on the target release. Only purls with matching tags are included as products. Purls without tags are always included. Vulnerabilities without tags are associated with all purls.

7.3. VEX Document Versioning

The OpenVEX version field is an incrementing integer. Since the CLI is stateless, the version is determined from the existing output file:

  1. If the output path points to an existing VEX file, parse the current version.

  2. New version = previous + 1.

  3. If no existing file, start at version 1.

8. HTML Report Generation

The primary output is a self-contained HTML file with a summary table. Each vulnerability entry is a single row.

name (if present) is displayed prominently. aliases shown as secondary identifiers. Tags are displayed for filtering context. comment is not included in HTML reports (it is internal-only metadata).

8.1. Detail Pages (Future)

A future version will generate per-vulnerability detail pages alongside the summary:

vulnlog-report/
  index.html                    # Summary table with links
  CVE-2021-44228.html           # Detail page
  ...

9. Validation Rules

  1. releases in vulnerability entries must reference IDs defined in releases.

  2. tags in vulnerability entries must reference IDs defined in tags (when tags section exists).

  3. tags in release purl entries must reference IDs defined in tags.

  4. fixed_in must reference a release ID defined in releases.

  5. reporter must be a valid reporter type enum value.

  6. justification is required when verdict is not_affected and forbidden otherwise.

  7. severity is required when verdict is affected or risk_acceptable.

  8. fixed_in is required when verdict is fixed and forbidden otherwise.

  9. source is required on report entries when reporter is other.

  10. All id fields within their respective sections must be unique.

  11. suppress.expires_at must be a valid future or present date at CLI execution time.

  12. analyzed_at, when present, must not be before the earliest at date across reports.

  13. Entries in releases must be in chronological order.

  14. All entries in packages must be valid Package URLs.

10. CLI Design

10.1. Usage

vulnlog <command> <file> [flags]

Every command except init requires a Vulnlog file as a positional argument. No automatic file detection.

10.2. Global Flags

Flag Description

-q, --quiet

Suppress non-essential output.

-v, --verbose

Verbose output.

--no-color

Disable colored output.

--format <f>

Output format for structured output: yaml (default). Future: json, text.

--version

Print version and exit.

--help

Print help.

10.3. Filtering Flags

Available on stats, report, suppress, and vex commands.

Flag Description

--release <id>

Include vulnerabilities affecting all releases up to and including this release.

--tags <id>

Filter by tag (repeatable).

10.4. Commands

10.4.1. vulnlog init

Scaffolds a minimal Vulnlog file containing only the required sections (version, project, releases, vulnerabilities).

vulnlog init [flags]
Flag Description

-o, --output <path>

Output file path (default: vulnlog.yaml).

--project <n>

Project name. Prompted if not provided.

--organization <n>

Organization name. Prompted if not provided.

10.4.2. vulnlog validate

vulnlog validate <file> [--strict]

--strict treats warnings as errors (e.g., missing verdict, missing description).

10.4.3. vulnlog lint

vulnlog lint <file> [--fix]

Enforces:

Top-level section order

version, project, tags, releases, vulnerabilities.

Vulnerability field order

id, name, aliases, releases, description, packages, reports, tags, analysis, analyzed_at, comment, severity, verdict, justification, fixed_in, fixed_at, fix_ref, fix_note.

Release chronological order

Oldest first.

10.4.4. vulnlog stats

vulnlog stats <file> [filter flags]

Example output:

project: Demo Project
total: 7
by_verdict:
  affected: 0
  not_affected: 2
  fixed: 2
  risk_acceptable: 1
  under_investigation: 2
by_severity:
  critical: 2
  high: 0
  medium: 0
  low: 2
  unrated: 1
suppressions:
  active: 2
  expired: 1

10.4.5. vulnlog report

vulnlog report <file> [filter flags] [--format html] [-o <path>]

Default output: vulnlog-report.html.

10.4.6. vulnlog suppress

vulnlog suppress <file> [--reporter <type>] [filter flags] [-o <path>]

When --reporter is specified, generates a single suppression file. When omitted, generates for all reporter types found in the vulnerability reports.

Output path: --output flag or default for the reporter type. --output is only valid with --reporter.

10.4.7. vulnlog vex

vulnlog vex <file> [filter flags] [-o <path>]

Default output: vex.json.

10.4.8. vulnlog sarif (Future)

Generates a SARIF document from Vulnlog data.

vulnlog sarif <file> [filter flags] [-o <path>]

Default output: vulnlog.sarif.json.

10.5. Exit Codes

Code Name Description

0

Success

Command completed successfully.

1

General error

Unexpected error (I/O failure, unhandled exception).

2

Validation error

The Vulnlog file contains validation errors.

3

File not found

The specified Vulnlog file does not exist.

4

Lint violation

The file has lint violations (without --fix).

5

Missing required flag

A required flag was not provided.

6

Invalid flag value

A flag references an unknown entity or unsupported type.

12. Examples

12.1. Quick Start

version: "1"

project:
  organization: Acme Corp
  name: Acme Web App

releases:
  - id: 1.0.0
    published_at: 2026-01-15

vulnerabilities:

  - id: CVE-2026-1234
    releases: [ 1.0.0 ]
    description: Remote code execution in example-lib
    packages:
      - "pkg:npm/example-lib@2.3.0"
    reports:
      - reporter: trivy
        at: 2026-01-20
    analysis: >
      The vulnerable code path is not reachable in our application
      because we only use the safe subset of the API.
    verdict: not_affected
    justification: vulnerable_code_not_in_execute_path

12.2. Full Example

version: "1"

project:
  organization: Demo Project Inc
  name: Demo Project
  security_team: Demo Security Team
  security_team_email: security@demo.com

tags:
  - id: binary
    description: Demo Project binary file
  - id: container
    description: Demo Project container image
  - id: sdk
    description: Demo Project SDK
  - id: build-dep
    description: Internal build dependency
  - id: pci
    description: PCI DSS compliance scope

releases:
  - id: 8.1.0
    published_at: 2021-01-15
    purls:
      - purl: "pkg:generic/demo-project@8.1.0"
        tags: [ binary ]
      - purl: "pkg:docker/demo-project@8.1.0"
        tags: [ container ]
  - id: 8.1.1
    published_at: 2021-02-11
    purls:
      - purl: "pkg:generic/demo-project@8.1.1"
        tags: [ binary ]
      - purl: "pkg:docker/demo-project@8.1.1"
        tags: [ container ]
  - id: 8.1.2

vulnerabilities:

  # No verdict - under investigation, with temporary suppression
  - id: CVE-2020-099
    releases: [ 8.1.0 ]
    description: Path traversal vulnerability in random-dependency
    packages:
      - "pkg:maven/org.random/random-core@1.33.7"
    reports:
      - reporter: trivy
        at: 2020-12-28
        suppress:
          expires_at: 2021-01-02
    tags: [ build-dep ]
    analysis: >
      Vulnerability in internal build dependency.
      No fixed library available at the moment. Re-check later.
    comment: >
      Checked upstream issue tracker - fix expected in random-core 1.33.8.
      Monitor https://github.com/random/random-core/issues/421
    severity: low

  # Stub entry - quick triage, no description, no verdict yet
  - id: CVE-2021-007
    releases: [ 8.1.1 ]
    packages:
      - "pkg:maven/com.example/some-lib@4.0.0"
    reports:
      - reporter: trivy
        at: 2021-03-10
    tags: [ binary ]

  # Verdict: not_affected - vulnerability does not impact this project
  - id: CVE-2021-002
    aliases: [ GHSA-xxxx-yyyy ]
    releases: [ 8.1.1 ]
    description: Deserialization vulnerability in some-dependency
    packages:
      - "pkg:maven/example.org/some-dependency@1.1.2"
    reports:
      - reporter: trivy
        at: 2021-03-03
      - reporter: snyk
        at: 2021-03-03
        vuln_id: [ SNYK-2021-002 ]
    tags: [ binary, container ]
    analysis: >
      Demo project does not utilize some-dependency
      and therefore is not affected.
    analyzed_at: 2021-03-03
    verdict: not_affected
    justification: vulnerable_code_not_present

  # Verdict: fixed - resolved in a subsequent release (named vulnerability)
  - id: CVE-2021-44228
    name: Log4Shell
    aliases: [ GHSA-jfh8-c2jp-5v3q ]
    releases: [ 8.1.1 ]
    description: RCE vulnerability in Log4J via JNDI lookup
    packages:
      - "pkg:maven/org.apache.logging.log4j/log4j-core@2.14.1"
    reports:
      - reporter: trivy
        at: 2021-12-10
    tags: [ binary, pci ]
    analysis: >
      Demo project utilizes Log4J and a PoC showed that
      the project is vulnerable to this RCE vulnerability.
    analyzed_at: 2021-12-10
    comment: >
      PoC exploit confirmed by Jane (security team) on 2021-12-10.
      See internal report attached to SEC-42.
    severity: critical
    verdict: fixed
    fixed_in: 8.1.2
    fixed_at: 2021-12-12
    fix_ref: "https://jira.example.com/browse/SEC-42"
    fix_note: "Updated Log4J from 2.14.1 to 2.17.1"

  # Verdict: not_affected - OS-level vulnerability in container base image
  - id: CVE-2024-5535
    releases: [ 8.1.1 ]
    description: Buffer overread in OpenSSL SSL_select_next_proto
    packages:
      - "pkg:apk/alpine/openssl@3.1.4-r0"
    reports:
      - reporter: trivy
        at: 2024-06-28
    tags: [ container ]
    analysis: >
      Vulnerable OpenSSL is present in the container base image
      but the affected NPN/ALPN negotiation code path is never
      invoked by our application.
    analyzed_at: 2024-06-29
    verdict: not_affected
    justification: vulnerable_code_not_in_execute_path

  # Verdict: risk_acceptable - vulnerability present but risk accepted
  - id: SNYK-2021-003
    releases: [ 8.1.2 ]
    description: DoS vulnerability in some-other-dependency
    packages:
      - "pkg:maven/com.some.other/some-other-dependency@3.11.0"
    reports:
      - reporter: snyk
        at: 2022-04-28
    tags: [ binary, container ]
    analysis: >
      Demo project utilizes some-other-dependency but only in rare cases.
      Therefore the risk is considered low and a fix would potentially
      be breaking for customer setups.
    analyzed_at: 2022-04-29
    comment: >
      Risk accepted by project lead (decision documented in RISK-12).
      Revisit when some-other-dependency 4.0 is released.
    severity: low
    verdict: risk_acceptable

  # Report from a non-scanner source (Oracle CPU)
  - id: CVE-2022-21449
    name: Psychic Signatures
    releases: [ 8.1.2 ]
    description: ECDSA signature verification bypass in Java SE
    packages:
      - "pkg:maven/com.oracle/jdk@17.0.2"
    reports:
      - reporter: other
        at: 2022-04-19
        source: "Oracle Critical Patch Update April 2022"
    tags: [ binary, container, pci ]
    analysis: >
      The project uses Java 17 and ECDSA signatures for
      JWT token verification. Confirmed exploitable.
    analyzed_at: 2022-04-20
    severity: critical
    verdict: fixed
    fixed_in: 8.1.2
    fixed_at: 2022-04-22
    fix_ref: "https://jira.example.com/browse/SEC-55"
    fix_note: "Updated JDK from 17.0.2 to 17.0.3"
