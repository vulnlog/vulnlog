== Purpose

This glossary defines the domain language of Vulnlog.
It is intended for developers, contributors, and users who want to understand the specific terms used in the Vulnlog project, its specification, documentation, and CLI.

== Core Concepts

Vulnlog::
An open-source tool and format for tracking, analyzing, and managing software vulnerabilities reported by SCA scanners.
The name combines "vulnerability" and "log."
The term refers to both the YAML file format and the CLI tool collectively.

Vulnlog file::
A YAML file following the Vulnlog specification, identified by the `.vl.yaml` (or `.vl.yml`) file extension.
Simple projects may use `vulnlog.yaml` (or `vulnlog.yml`).
Each Vulnlog file is scoped to a single release branch.
Example: `demo-project-8.1.vl.yaml`.

Vulnlog CLI::
The command-line interface tool that reads Vulnlog files and generates output artifacts (suppression files, HTML reports, VEX documents).

== Vulnerability Tracking

Vulnerability entry::
A single record in the `vulnerabilities` section of a Vulnlog file.
Captures a finding (what was found, where, by whom) and the human verdict about it.
Vulnerability entries are ordered newest first.

Vulnerability ID::
The primary identifier of a vulnerability entry, stored in the `id` field.
Must be unique across all entries and aliases within the file.
Preferably a CVE identifier (e.g., `CVE-2021-44228`).

Vulnerability name::
An optional common name for a well-known vulnerability (e.g., "Log4Shell", "Heartbleed").
Stored in the `name` field.

Alias::
An alternative identifier for the same vulnerability, stored in the `aliases` array.
Must not overlap with any `id` or `aliases` in other entries within the same file.
Used for cross-referencing in reports but not for suppression file generation.

Verdict::
The human triage decision recorded for a vulnerability entry, stored in the `verdict` field.
The term "verdict" reflects a deliberate judgment by an engineer, distinct from automated scanner statuses.
Possible values: `affected`, `not_affected`, `risk_acceptable`.
When absent, the vulnerability is considered under investigation.
"Fixed" is not a verdict -- it is a resolution action recorded separately.

Under investigation::
The implicit state of a vulnerability entry when no `verdict` has been set.
Maps to `under_investigation` in OpenVEX output.

Analysis::
The free-text rationale for the triage decision, stored in the `analysis` field.
Records *why* the team reached their verdict.
Used as `impact_statement` in VEX output for `not_affected` entries and as `action_statement` for `risk_acceptable` entries.

Comment::
Additional information not related to the analysis itself, stored in the `comment` field.
Not used in any output generation.

Resolution::
A structured record that a dependency update or other corrective action was taken, stored in the `resolution` block.
Independent of the verdict -- a vulnerability may be `not_affected` (verdict) yet still have a resolution (the dependency was updated defensively).
When `verdict` is `affected` and a `resolution` is present, the combination maps to VEX status `fixed`.

Severity::
The assessed severity of a vulnerability.
Values: `critical`, `high`, `medium`, `low`.
Required when the verdict is `affected` or `risk_acceptable`.

Justification::
The reason why a vulnerability does not affect the project, stored in the `justification` field.
Required when the verdict is `not_affected`.
Values align with the OpenVEX justification vocabulary.

Stub entry::
A minimal vulnerability entry containing only required fields (`id`, `releases`, `packages`, `reports`).
Created during quick triage using the `vulnlog add` command.

== Project Structure

Project::
The `project` section of a Vulnlog file.
Contains metadata: organization, project name, author, and optional contact.

Tag::
A label defined in the optional `tags` section.
Provides flexible categorization for deliverables, usage contexts, or any team-specific taxonomy.

Release::
A software release tracked in the `releases` section.
Releases must be listed in chronological order (oldest first) because the CLI uses array position for range resolution.

Release branch::
A line of releases sharing a common major/minor version.
Each Vulnlog file is scoped to a single release branch.

== Package and Dependency Concepts

Package::
An affected dependency listed in the `packages` array as a Package URL string.

Package URL (purl)::
A standardized format for identifying software packages.
Examples: `pkg:maven/org.apache.logging.log4j/log4j-core@2.14.1`, `pkg:docker/demo-project@8.1.1`.

Release purl::
A versioned Package URL in a release's `purls` section.
Identifies a release artifact and optionally carries tags for scoped VEX generation.
Maps to "products" in OpenVEX.

== Report and Triage Concepts

Report (vulnerability report)::
A record in the `reports` array indicating that a specific scanner or source reported the finding.
Not to be confused with HTML report (an output artifact).

Reporter type::
The `reporter` field on a report entry, identifying the scanner or source.
A fixed enum (e.g., `trivy`, `snyk`, `dependency_check`, `other`).
Determines the suppression file format.

Source::
The `source` field on a report entry for non-scanner reports.
Required for reporter type `other`.

Suppression::
A directive to exclude a vulnerability from a scanner's findings.
Can be temporary (with `expires_at`) or permanent (without `expires_at`).
Stored in the `suppress` block on a report entry.

== Output Artifacts

Suppression file::
A file generated by the Vulnlog CLI that instructs an SCA scanner to suppress specific findings.
Each scanner has its own format.
Vulnerabilities with `verdict: affected` and a `resolution` are never suppressed.

HTML report::
An HTML file for communicating vulnerability analysis to stakeholders.
Each vulnerability entry corresponds to a single row.

VEX document::
An OpenVEX document communicating whether vulnerabilities are exploitable.
The CLI manages document versioning by reading the previous output and incrementing the version counter.

OpenVEX::
The VEX output format used by Vulnlog.
Key mapping: `affected` + `resolution` = VEX `fixed`. `not_affected` = VEX `not_affected`. `risk_acceptable` = VEX `affected` with action statement.

== Validation and Linting

Validation::
Checking a Vulnlog file for structural correctness and semantic integrity, including unique ID enforcement.

Linting::
Checking for consistent formatting and field ordering.
Releases: oldest first. Vulnerabilities: newest first.

Release range::
`--release <id>` means "all releases up to and including this release," resolved by array position.
